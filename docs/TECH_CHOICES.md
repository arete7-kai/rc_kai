# 技术选型说明（TECH CHOICES）

> 本文回答作业"其他说明"里的两个问题——**为什么选它 / 不用时的替代方案**——并逐项落到每个技术组件上。
>
> 关于"为什么用数据库当队列、而不用消息中间件"这条，已在 `README.md` §9 单独展开，本文不重复，只在 §6 指回。
>
> 贯穿全文的一条准绳：**这是 MVP，选型偏向"依赖少、可复现、把关键逻辑显式暴露出来"，而不是"技术先进 / 功能齐全"。**

---

## 选型总览

| 组件 | 选择 | 主要备选 | 一句话理由 |
|---|---|---|---|
| 语言 | Go | Python/FastAPI、Node/TS、Java/Spring | IO 密集的并发中继场景，goroutine + 单二进制最契合 |
| 存储 / 队列 | PostgreSQL | MySQL、SQLite、专用 broker | `SKIP LOCKED` 让一个库同时当持久层和并发队列 |
| HTTP / 路由 | 标准库 `net/http` | Gin、Echo、Chi | 三个端点，框架是纯负担 |
| 数据访问 | `database/sql` + 原生 SQL | GORM、sqlx、ent | 要把 `SKIP LOCKED` 那条查询**显式暴露**，ORM 反而遮住它 |
| 环境 | Docker Compose 起 Postgres | 本地装库、托管 DB | 评审一条命令即可复现 |
| 日志 | 标准库 `log/slog` | zap、logrus | 结构化日志够用，不引第三方 |
| 中间件 | 无（DB-as-queue） | Kafka/RabbitMQ/Redis | 见 README §9 |

---

## 1. 语言：Go

**为什么选它。** 这个系统的本质是"大量并发的出站 HTTP 调用 + 长期运行的后台 worker"，是典型的 IO 密集型网络服务。Go 在这几点上正好对上：

- **并发模型天然契合**：goroutine + worker 池就是这个系统的核心结构，不需要额外的异步框架或线程池调优。
- **单静态二进制**：`go build` 出一个文件，部署即拷贝，契合 DECISIONS 里"运维零成本"的目标。
- **标准库够强**：`net/http`（收发都能用）、`database/sql`、`log/slog` 让第三方依赖可以压到接近零，减少要审计和会出事的表面积。
- **信号价值**：对这种基础设施类服务，Go 是行业里很主流的选择，能体现"这是给真实生产写的"。

**备选与为什么不选。**
- **Python / FastAPI**：出 MVP 更快、可读性好，但"几百并发出站请求"的并发故事偏弱（要靠 asyncio + 额外任务框架如 Celery），依赖反而更多。
- **Node / TypeScript**：本身没问题，但社区惯性会把人推向 BullMQ + Redis——而 Redis 正是我在 DECISIONS 里刻意不引入的，等于给自己埋一个"过度设计"的坑。
- **Java / Spring**：企业级但对一个 MVP 太重，启动、配置、样板代码成本都高，与"第一版要轻"冲突。

**接受的代价。** 比 Python 多一些样板代码；要求接手的人会 Go。在本项目规模下这点代价可接受。

## 2. 存储 / 队列：PostgreSQL

**为什么选它。** 存储选型的决定性因素不是"能不能存"，而是**能不能安全地兼任任务队列**——因为整个 v1 的核心赌注就是"DB 即队列"（README §9）。Postgres 在这件事上有几个别人替代不了的点：

- **`SELECT ... FOR UPDATE SKIP LOCKED`**：让多个 worker（乃至多进程副本）并发领取任务而互不抢占、无需外部分布式锁。这是"不上消息队列还能安全并发"的技术基石。
- **强事务保证**：支撑"先落盘再返回 202"这条可靠性地基（DECISIONS D1）。
- **JSONB**：不同供应商的 Header/Body 格式各异，用 JSONB 存任意结构，省掉建模负担。
- **部分索引 + `LISTEN/NOTIFY`**：给队列查询建 `(status, next_attempt_at)` 部分索引、用 NOTIFY 减少空轮询——这些正是 README §9 里"延长 DB-as-queue 跑道"的手段，都是 Postgres 原生能力。

**备选与为什么不选。**
- **SQLite**：跑 demo 很方便，但单写者、并发弱，无法真实展示"多 worker 用 SKIP LOCKED 领取"这套并发模型——而这恰恰是本设计最想证明的东西，所以放弃。
- **MySQL**：8.0 之后也有 `SKIP LOCKED`，技术上可行；但 Postgres 的 JSONB + 部分索引 + LISTEN/NOTIFY 这套组合对"DB 当队列"这个特定模式支持得更顺手、更成熟。
- **专用 broker（Kafka/RabbitMQ/Redis）**：即"用不用中间件"的问题，理由见 README §9——v1 明确不上。

**接受的代价。** 需要一个运行中的 Postgres 实例；用 Docker Compose 消化掉这点摩擦（见 §5）。

---

## 3. HTTP / 路由：标准库 `net/http`

**为什么选它。** MVP 对外只有三个端点（提交、查状态、重投死信）。`net/http` 直接够用，路由手写几行即可。

**备选与为什么不选。** Gin / Echo / Chi 等框架在这个规模下带来的是**净负担**：多一层要学的概念、多一个依赖、多一份升级维护，却换不来任何实际收益。等端点数量、中间件需求（鉴权、限流、metrics）真正长起来时再引入框架，是更诚实的时机。

**接受的代价。** 手写一点路由和参数解析样板；在三个端点的规模下可忽略。

---

## 4. 数据访问：`database/sql` + 原生 SQL（不用 ORM）

**为什么选它。** 这是一个**刻意**的决定，不只是省事：本系统最关键的一段逻辑就是 `FOR UPDATE SKIP LOCKED` 那条领取查询和状态流转的 UPDATE。用原生 SQL 能把它**显式、清楚地摆在代码里**，让人一眼看懂并发领取是怎么发生的。

**备选与为什么不选。** GORM / ent 这类 ORM 恰恰会把这条最需要被看见的查询藏进抽象里，还可能生成非预期的 SQL，与"要精确控制锁行为"的诉求相悖。sqlx 折中一点，但在本项目的 SQL 量级下，标准库已经够。

**接受的代价。** 手写 SQL 和行扫描代码；换来的是对关键并发语义的完全掌控，值得。

---

## 5. 环境：Docker Compose 起 Postgres

**为什么选它。** 一条 `docker compose up` 就能把 Postgres 拉起来，评审无需在自己机器上装库、配用户、建库，**开箱即复现**。这对一个要被别人 clone 下来跑的作业尤其重要。

**备选与为什么不选。** 让评审本地装 Postgres 或连托管 DB，都会引入"装不上 / 版本不对 / 连不上"的摩擦，直接降低作业被跑起来的概率。

**接受的代价。** 需要评审装了 Docker；这是目前最低摩擦的假设。

---

## 6. 中间件：无（数据库即队列）

这条是作业明确点名要说明的"是否使用中间件"。结论是 **v1 不引入任何消息中间件**，用 Postgres + `SKIP LOCKED` 承担队列职责；替代方案（也即现方案）、以及"何种信号出现才升级到 broker"，已在 **`README.md` §9** 和 **`docs/DECISIONS.md` D2** 完整展开，此处不重复。

---

## 一句话收束

选型的共同逻辑是**用最少的、最能把关键逻辑暴露出来的技术**去满足 MVP：Go 提供并发与低运维，Postgres 一件工具兼两职，其余一律标准库。每引入一个东西都要能回答"它现在解决的是真问题吗"——答不上来的（框架、ORM、中间件），就不加。
