# AI 使用说明（AI USAGE）

> 对应作业要求的三件事：AI 在哪些关键地方帮了忙、AI 给过哪些我没采纳的建议、哪些关键决策是我自己做的及原因。
>
> 记录方式是边做边往对应小节丢一行第一手散记，收尾时再润色成通顺段落（只润色、不替我回忆）。

---

## 一、AI 在哪些关键地方提供了帮助

**设计阶段**，AI 帮我把"可靠出站中继"这个问题的地基收敛成一句话——先落盘再确认、因而只能是 at-least-once；也提示了几个我原本没充分意识到的真实工程点：出站中继天然的 SSRF 风险、某供应商拖累全局的队头阻塞、以及"业务事务真正提交后才发通知"应归在业务方的事务性 Outbox 边界。规模判断上，它帮我推导了 DB-as-queue 的上限：一条通知成功路径上至少 3 次 DB 写，真正的天花板是 autovacuum 跟不上表膨胀，而不是 CPU。

**编码第 1 步（建表与存储）**，AI 落地了 migrations（destinations 注册表 + notifications 队列表）、docker-compose 与 internal/store，并在关键 SQL 上逐条标注对应的决策（D1 幂等入队、D2 的 SKIP LOCKED 领取、D4 死信重投、§4.7 租约回收）；它还主动指出 database/sql 必须搭配一个第三方驱动，并把这个依赖决定留给我确认，没有擅自 go get。

**编码第 2 步（worker 与 SSRF）**，AI 把 SSRF 防护做在了关键点上：不只校验 URL 里的 host 字符串，而是用 net.Dialer.Control 在建连前校验实际解析出的 IP，从而防住 DNS rebinding，并顺带禁掉 3xx 重定向跟随这一 SSRF 向量；它把 ErrBlockedIP 设计成可用 errors.Is 识别的哨兵错误，让 worker 能把"连接时命中内网"判为不可重试而直接进死信，而非盲目重试。错误分类与指数退避 + 抖动也一并落地，并逐行注释对应 README §4.5 / D4，退避取 [50%,100%) 区间抖动以避免供应商恢复瞬间的重试风暴。

**编码第 3 步（api 与进程装配）**，AI 实现了三个端点（纯 net/http 的方法 + 通配路由）与 cmd/relay/main.go 的装配（pgx、worker 池、租约回收循环、优雅关闭），配置全走环境变量、无硬编码密钥；它主动把入队从"live 回查注册表"改成"入队快照"（把 url/method/headers/secret_ref 与 max_attempts/backoff_cap 一并冻结进通知行），使在途任务行为可预测，与 D3 的 URL 快照是同一原则；GET 状态接口也被设计成"安全视图"，明确不返回 body / headers / secret_ref，避免密钥或负载经查询接口泄漏。

**测试与第 4 步（单测与跑通 demo）**，AI 把 worker 的错误分类从 deliver 内联 switch 抽成纯函数 classify 以便表驱动单测（覆盖 2xx/4xx/429/5xx/超时/连接错误，重点验证 429 与其它 4xx 相反），退避与 IP 黑名单也各配了表驱动测试；它还落地了可控失败的 mock upstream（ok/fail/reject/flaky/timeout，flaky 按 Idempotency-Key 分组计数）、seed 与 demo 脚本，并实测 mock 五种行为返回码正确；SSRF 白名单则实现为 host 级匹配、连接时也按原始 host 放行（避免 host↔IP 对不上），且只放行"私网/回环检查"这一条，scheme 与 D3 仍照走。

## 二、AI 给过但我没有采纳的建议

> 这一节最能体现"对 AI 输出的判断与取舍"。每条都是：AI 建议了什么 + 我为什么否。

**设计阶段**，AI 的默认方案里带了每目标熔断器 / 自适应限流，我砍掉了——v1 用退避 + 死信已能兜底，熔断是"某供应商开始拖累全局"之后才需要的隔离手段，推到 v2；它也倾向于引入消息中间件（Kafka/Redis），同样被我否掉——预期量级下 DB-as-queue 足够，过早引入是纯粹的复杂度负债，何时才上见 README §8。

**编码第 1 步**，AI 想直接 go get 一个 Postgres 驱动好让 store 包"能跑"，被 CLAUDE.md"加任何第三方库前先问我"拦下，改为只依赖标准库 database/sql 写 store（不导入驱动、go.mod 保持零依赖），驱动选型留到装配 main 时再定；它还一度想在租约回收（ReclaimExpired）时给 attempts +1 来"兜底防止崩溃循环"，被"克制优先、不加以防万一功能"的原则拦下，v1 回收不计尝试次数，只在代码注释里点明这个取舍（即下文 D6）。

**编码第 2 步**，AI 提议在 worker 里给每个 destination 加独立并发上限 / 熔断来隔离坏邻居，被"不加熔断器、自适应限流（v2 才考虑）"拦下；它还提议给投递加"可配置的重定向跟随白名单"，被"克制优先"拦下——v1 直接禁止跟随 3xx，更安全也更简单。

**编码第 3 步**，AI 一开始让在途 deliver 直接复用 shutdown 的 ctx，那样一收到关闭信号就会立即掐断在途投递；我要求改成"在途用独立 ctx 自然收尾 + 一个 SHUTDOWN_GRACE 总上限兜底"，避免为一个挂死的上游等到天荒地老。它还把单次投递的整体超时默认设成 30s，我压到 10s——过长会让慢上游长时间占住 worker，是队头阻塞的雏形。

**编码第 4 步**，面对"SSRF 拦截挡掉本地 mock"这个矛盾，AI 把"用一个开关整体关闭 SSRF 检查"也列为可选项，我否掉了这个方向——安全设计应"默认拒绝 + 显式白名单"，更不该让一个配置项去动安全地基本身，最终改为只放行"私网检查"这一条的 host 白名单（SSRF_ALLOW_HOSTS，默认空即严格）；测试上 AI 想顺手补几条 store/api 的集成测试与 e2e，我按"只测三个纯函数、不追覆盖率、不测集成链路"的要求砍掉，集成靠 demo 脚本肉眼验收即可。

## 三、哪些关键决策是我自己做出的，以及原因

> 对应 docs/DECISIONS.md 的 D1–D6，用自己的话概括"我定了什么、为什么"。

- **D1 投递语义 = at-least-once**：不做 exactly-once，因为跨第三方根本做不到（需接收方全部支持幂等键，不受我控制）；我明确接受"会重复"的代价，靠接收方幂等 + 透传幂等键兜底。

- **D2 v1 用 DB-as-queue、不上中间件**：基于我自己给的规模假设（峰值 ingest ≤500/s、供应商 ≤数十、扇出 ≤1:数个），并列出了升级到 broker 的具体指标信号，而不是凭感觉换。

- **D3 仅注册表、不开放任意 URL**：为集中管理密钥 + 天然形成 SSRF 白名单，我接受"接入方多一步登记"的代价，也不留"任意 URL 兜底"的口子。

- **D4 分级重试 + 死信 ≠ 日志**：按 destination 打严重等级决定重试上限与告警；死信是可查询、可一键重投的结构化记录，而不是只写日志。

- **D5 顺序**：用"同 key + 可交换性"两条判据重核了三个场景（并修正了最初的直觉误判——状态赋值类才是高危区）；v1 不实现 FIFO，只在注册表标注需求，再用 payload 版本号 / last-write-wins 兜底，真正的 per-key FIFO 推到 v2。

- **D6 崩溃回收不计入 attempts**：租约超时回收把卡住的 DELIVERING 行重置为 PENDING 时，不给 attempts +1——那次崩溃并没有产生一次完整的投递尝试结果，计入只会让正常的重启/部署白白消耗重试预算；代价是理论上的重复投递，与 D1 的 at-least-once 一脉相承、由接收方幂等兜底，若日后观测到某任务反复被回收，再引入独立的 reclaim_count 上限来止损。
