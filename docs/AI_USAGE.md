# AI 使用说明（AI USAGE）

> **我怎么用 AI**：这个项目里，AI 承担了两个角色——**分析者**（把问题讲透、把利弊摆全、给出可选项与推导）和**执行者**（按我定的边界写代码）。我承担的是**判断者与取舍者**：读懂分析、决定哪些采纳哪些拒绝、为每个取舍的后果负责，并在动手前把边界写成硬约束（`CLAUDE.md`）约束 AI 不越界。本项目的一些技术洞见（SSRF、队头阻塞、快照 vs live、崩溃回收语义）由 AI 先提出，经分析后我判断采纳；我认为**关键不在点子是否原创，而在能否判断该不该采纳、并为后果负责**。下面如实区分哪些是我的原生判断、哪些是我被说服后的取舍。

---

## 一、AI 在哪些关键地方提供了帮助

**问题拆解与边界界定**：AI 帮我把"可靠出站中继"这个模糊需求收敛成一条可执行的地基——**先落盘再返回 202，因而语义只能是 at-least-once**；并提示了几个我原本没意识到的真实工程风险点：出站中继天然是一个 **SSRF 代理**、某供应商长期不可用会造成**队头阻塞**、以及"业务事务提交后才发通知"应归属**业务方的事务性 Outbox** 而非本服务。这些提示扩大了我判断边界时的视野。

**复杂度的量化推导**：在"要不要上消息队列"这个核心取舍上，AI 帮我把直觉变成了可辩护的量化依据——一条通知成功路径至少 3 次 DB 写、`SKIP LOCKED` 的正确性与供应商数量无关、DB-as-queue 真正的天花板是 autovacuum 跟不上表膨胀而非 CPU，以及一组具体的"何时才升级到 broker"的指标信号。

**编码执行（按我定的边界）**：AI 逐层落地了 migrations、`internal/store`（`SKIP LOCKED` 领取 + 租约回收）、`internal/worker`（错误分类 + 指数退避抖动）、`internal/security`（SSRF）、`internal/api`（三端点）与 `cmd/relay`（装配 + 优雅关闭），关键 SQL/逻辑都按要求逐条标注对应的决策编号。其中几处执行质量值得记下：SSRF 校验做在 `net.Dialer.Control` 层、针对**实际解析出的 IP**（防 DNS rebinding）而非 URL 字符串；`ErrBlockedIP` 设计成 `errors.Is` 可识别的哨兵错误；错误分类抽成纯函数 `classify` 以便表驱动单测。

**验证支撑**：AI 落地了可控失败的 mock upstream（ok/fail/reject/flaky/timeout，flaky 按幂等键分组计数）、seed 与 demo 脚本，使我能端到端跑出"成功 / 重试后成功 / 重试到上限进死信"的完整全景并留证。

## 二、AI 给过但我没有采纳的建议


**我一开始就拒绝、不需要被说服的：直接上消息中间件。** 项目起步时 AI 的默认方案倾向引入 Kafka/Redis 这类中间件。我直接否了——一个第一版系统就上重型 broker，是我判断中典型的"用复杂度换想象中的规模"。我的立场是：先证明 DB-as-queue 在预期量级下不够用，再升级；在此之前引入 broker 纯属复杂度负债。这条我在动手前就写进了 `CLAUDE.md` 的禁止清单，让 AI 后续不再反复提议。

**我识别并纠正的 AI 错误：`lib/pq` 的选型理由。** 在 Postgres 驱动选型上，AI 一度暗示 `lib/pq` "更贴合显式暴露 SQL"。我判断这个理由**站不住**——`pgx/v5/stdlib` 走标准 `database/sql` 接口时，原生 SQL 的写法一行都不用改，"显式 SQL"两者都满足，真正的差异只是维护活跃度与性能。我据此纠正了选型,改用维护更活跃、对 `LISTEN/NOTIFY` 支持更好的 `pgx`。这是一次我对 AI 输出的事实性纠错，而非风格偏好。

**经分析后我判断"该拒绝"的过度设计**（这些是 AI 提出、我评估利弊后决定不采纳的取舍，判断依据是"它解决的是 v1 的真问题还是更大规模才有的问题"）：
- **每目标熔断器 / 自适应限流**：是好东西，但属于"某供应商开始拖累全局"之后才需要的隔离手段；v1 用退避 + 死信已能兜底，我判断推到 v2。
- **插件化 payload 转换引擎**：业务方直接发最终 payload 即可，我判断这是臆想需求。
- **可配置的 3xx 重定向跟随白名单**：我选更简单也更安全的"直接禁止跟随 3xx"。
- **store/api 的集成测试与 e2e**：我判断集成行为用 demo 脚本肉眼验收更划算，单测只覆盖三个纯函数（退避 / 错误分类 / SSRF 黑名单），不追覆盖率。

**经分析后我判断"该做"、哪怕 AI 起初没强调的**：
- **SSRF 防护纳入 v1**：AI 提示了这个风险后，我判断"一个能对任意 URL 发请求的内部服务本质上就是一个可被利用的 SSRF 代理"是真实的生产级隐患，而不是过度设计——所以我拍板把它放进 v1，而不是留到以后。
- **优雅关闭必须给在途投递独立 ctx + 总超时上限**：AI 最初让在途 `deliver` 直接复用 shutdown 的 ctx，那样一收到关闭信号就会掐断在途投递、制造"结果未知"的投递。我要求改成"在途用独立 ctx 自然收尾 + `SHUTDOWN_GRACE` 总上限兜底"，避免为一个挂死的上游无限等待。
- **投递整体超时压到 10s**：AI 默认 30s，我判断过长会让慢上游长时间占住 worker（队头阻塞的雏形），压到 10s。

**安全设计上我坚持的原则：SSRF 例外用白名单，不用总开关。** demo 的 mock 跑在回环地址会被 SSRF 拦截，AI 把"一个开关整体关闭 SSRF 检查"列为可选项。我否掉了这个方向——安全设计应"默认拒绝 + 显式白名单"，绝不该让一个配置项去动安全地基本身。最终改为 `SSRF_ALLOW_HOSTS`（默认空即严格），且只放行"私网/回环检查"这一条，scheme 校验与"目标必须来自注册表"仍照走，白名单不是绕过一切的后门。

## 三、哪些关键决策是我自己做出的，以及原因

以下六个决定系统本质的取舍由我拍板，并标注判断来源：

- **D1 投递语义 = at-least-once**：不做 exactly-once，因为跨第三方根本做不到（需接收方全部支持幂等键，不受我控制）；我明确接受"会重复"的代价，靠接收方幂等 + 透传幂等键兜底。会重复的具体情形：投递已成功、但写回 `DELIVERED` 前进程崩溃，租约回收后会再投一次。

- **D2 v1 用 DB-as-queue、不上中间件（我最坚持的原生判断）**：基于我给的规模假设（峰值 ingest ≤500/s、供应商 ≤数十、扇出 ≤1:数个），此量级下单实例 Postgres + `SKIP LOCKED` 成立；并列出了升级到 broker 的具体指标信号，而不是凭感觉换。

- **D3 仅注册表、不开放任意 URL**：为集中管理密钥 + 天然形成 SSRF 白名单，我接受"接入方多一步登记"的代价，也不留"任意 URL 兜底"的口子——留了口子，SSRF 白名单和密钥集中的好处就漏掉一半。

- **D4 分级重试 + 死信 ≠ 日志**：按 destination 打严重等级决定重试上限与告警；死信是可查询、可一键重投的结构化记录，而不是只写日志。

- **D5 顺序**：用"同 key + 可交换性"两条判据重核三个场景，并**修正了我最初的直觉误判**——一开始我凭业务名称以为"付款安全、库存不安全"，用判据重核后发现基本反了：状态赋值类（付款改状态）才是顺序敏感的高危区，增量式库存反而安全。v1 不实现 FIFO，只在注册表标注需求，再用 payload 版本号 / last-write-wins 兜底，真正的 per-key FIFO 推到 v2。

- **D6 崩溃回收不计入 attempts**：租约超时把卡住的 `DELIVERING` 行重置为 `PENDING` 时不给 `attempts +1`——那次崩溃并未产生一次完整的投递尝试结果，计入只会让正常的重启/部署白白消耗重试预算。代价是理论上的重复投递（概率极低，需连续在同一窄窗口崩溃），与 D1 一脉相承、靠幂等兜底；若日后观测到某任务反复被回收，再引入独立的 `reclaim_count` 上限止损，而不是现在就加。

## 我的 AI 工作流

核心是**始终把 AI 当分析者和执行者，判断权留在自己手里**：

1. **设计先行，决策与编码分离**：先出设计文档、再把关键取舍逐条拍板（D1–D6），确认边界后才开始写代码。不在需求还模糊时就让 AI 生成实现。

2. **边界前置成硬约束**：动手前把"不做什么"写进 `CLAUDE.md`（不上中间件、不加熔断、不拆微服务、不开放任意 URL 等），让 AI 每次触碰这些边界都被拦下。这把"防止 AI 好心办坏事"从"每次盯着"变成了"一次写死"——事实证明它拦下了多次过度设计的提议。

3. **小步走，每步人工 review 才放行**：一层一层推进（store → worker/security → api → mock/demo），每完成一层我先跑起来验证、再决定是否进入下一层，不一口气生成整个项目。层与层之间先对齐接口再写实现，避免最贵的返工。

4. **关键决策自己拍板，AI 只执行**：每个影响系统本质的取舍都由我做决定，AI 负责把利弊分析清楚和落地编码。遇到 AI 提议超出边界，我以"违反 CLAUDE.md"直接拦回。

5. **边做边记 AI 使用痕迹**：每步把"AI 帮了什么、我否了什么、为什么"当场记一行，最后再润色，避免事后回忆把 AI 的建议误记成我的决定。
