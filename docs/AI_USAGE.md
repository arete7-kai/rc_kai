# AI 使用说明（AI USAGE）

> 对应作业要求的三件事：AI 在哪些关键地方帮了忙、AI 给过哪些我没采纳的建议、哪些关键决策是我自己做的及原因。
>
> 用法：**边做边往对应小节丢一行**，第一手记录最可信。收尾时再让 AI 把散句润色成通顺段落即可（只润色、不替我回忆）。
> 格式：`- [日期/阶段] 事件 —— （理由/结果）`

## 一、AI 在哪些关键地方提供了帮助

- [设计阶段] AI 提示了几个我原本没意识到的真实工程点：SSRF 风险、队头阻塞、事务性 Outbox 属于业务方边界。
- [设计阶段] AI 帮我推导了 DB-as-queue 的规模上限（写放大 3×、天花板是 autovacuum 而非 CPU）。
- [编码阶段] 
- [编码阶段] 
- [排障阶段] 
- [编码阶段·第1步] AI 落地了 migrations（destinations 注册表 + notifications 队列表）、docker-compose、internal/store 三件，并在关键 SQL 上逐条标注对应的 DECISIONS（D1 幂等入队 / D2 SKIP LOCKED 领取 / D4 死信重投 / §4.7 租约回收）。
- [编码阶段·第1步] AI 主动指出 database/sql 必须搭配一个第三方驱动（lib/pq 或 pgx），并按项目约束把这个依赖决定留给我确认，没有擅自 go get。
- [编码阶段·第2步] AI 把 SSRF 防护做对了关键点：不只校验 URL 里的 host 字符串，而是用 net.Dialer.Control 在建连前校验“实际解析出的 IP”，从而防住 DNS rebinding；并顺带禁掉了 3xx 重定向跟随（也是 SSRF 向量）。
- [编码阶段·第2步] AI 把 ErrBlockedIP 设计成可被 errors.Is 识别的哨兵错误，让 worker 能把“连接时命中内网”判为不可重试（直接死信）而非盲目重试。
- [编码阶段·第2步] AI 落地了错误分类与指数退避+抖动，并逐行注释对应 README §4.5 / D4；退避用 [50%,100%) 抖动避免恢复瞬间的重试风暴。
- [编码阶段·第3步] AI 落地了 internal/api（三端点，纯 net/http 的方法+通配路由）与 cmd/relay/main.go（pgx 装配 + worker 池 + 租约回收循环 + 优雅关闭）；配置全走环境变量、无硬编码密钥。
- [编码阶段·第3步] AI 主动把入队从“live 回查注册表”改成“入队快照”（url/method/headers/secret_ref + max_attempts/backoff_cap 一并冻结进通知行），保证在途任务行为可预测，与 D3 的 URL 快照同一原则。
- [编码阶段·第3步] AI 把 GET 状态接口设计成“安全视图”，明确不返回 body / headers / secret_ref，避免密钥/负载经查询接口泄漏。
- [测试阶段·第4步] AI 把 worker 的错误分类从 deliver 里内联 switch 抽成纯函数 classify，便于表驱动单测（覆盖 2xx/4xx/429/5xx/超时/连接错误，重点验证 429 与其它 4xx 相反）；退避与 IP 黑名单也各配了表驱动测试。
- [编码阶段·第4步] AI 落地了可控失败的 mock upstream（ok/fail/reject/flaky/timeout，flaky 按 Idempotency-Key 分组计数）+ seed（三个 destination）+ demo 脚本，并实测 mock 五种行为返回码正确。
- [编码阶段·第4步] AI 把 SSRF 白名单实现成 host 级匹配、连接时也按原始 host 放行（避免 host↔IP 对不上），且只放行“私网/回环检查”这一条，scheme 与 D3 仍照走。

## 二、AI 给过但我没有采纳的建议

> 这一节最能体现"对 AI 输出的判断与取舍"，是评分重点。每条写：AI 建议了什么 + 我为什么否。

- [设计阶段] AI 的默认方案里带了熔断器/自适应限流，我砍掉——v1 用退避+死信已能兜底，熔断是"某供应商拖累全局"之后才需要的隔离手段（推到 v2）。
- [设计阶段] AI 倾向于引入消息中间件（Kafka/Redis），我否了——预期量级下 DB-as-queue 足够，过早引入是复杂度负债（何时才上见 README §8）。
- [编码阶段] 
- [编码阶段] 
- [编码阶段·第1步] AI 想直接 `go get` 一个 Postgres 驱动来让 store 包“能跑”，被 CLAUDE.md“加任何第三方库前先问我”拦下——改为只依赖标准库 database/sql 写 store（不导入驱动、go.mod 保持零依赖），驱动选型留到装配 main 时再定。
- [编码阶段·第1步] AI 一度想在租约回收（ReclaimExpired）时给 attempts +1 以“兜底防止崩溃循环”，被“克制优先、不加以防万一功能”的原则拦下——v1 回收不计尝试次数，只在代码注释里点明这个取舍。
- [编码阶段·第2步] AI 提议在 worker 里对每个 destination 加“独立并发上限 / 熔断”隔离坏邻居，被 CLAUDE.md“不加熔断器、自适应限流（v2 才考虑）”拦下——v1 靠退避+死信兜底即可。
- [编码阶段·第2步] AI 提议给投递加“可配置的重定向跟随白名单”，被“克制优先”拦下——v1 直接禁跟随 3xx（更安全也更简单），有需要再说。
- [编码阶段·第3步] AI 一开始让在途 deliver 直接复用 shutdown ctx，会在关闭时立即掐断在途投递；我要求改为“在途用独立 ctx 自然收尾 + 一个 SHUTDOWN_GRACE 总上限兜底”，避免等挂死上游等到天荒地老。
- [编码阶段·第3步] AI 把单次投递整体超时默认设成 30s，我压到 10s——过长会让慢上游长时间占住 worker（队头阻塞雏形）。
- [编码阶段·第4步] 面对“SSRF 拦截挡掉本地 mock”这个矛盾，AI 把“一个开关整体关闭 SSRF 检查”也列为可选项；我否了这个方向——安全设计应“默认拒绝 + 显式白名单”，且不该让配置改动去动安全地基本身，最终改为只放行“私网检查”这一条的 host 白名单（SSRF_ALLOW_HOSTS，默认空）。
- [测试阶段·第4步] AI 想顺手补几条 store/api 的集成测试与 e2e，我按“只测三个纯函数、不追覆盖率、不测集成链路”的要求砍掉——集成靠 demo 脚本肉眼验收即可。

> 对应 docs/DECISIONS.md 的 D1–D5，用自己的话概括"我定了什么、为什么"。

- **D1 投递语义 = at-least-once**：不做 exactly-once，因为跨第三方做不到（需接收方全支持幂等键，不受我控制）；明确接受"会重复"的代价，靠接收方幂等+透传幂等键兜底。
- **D2 v1 用 DB-as-queue、不上中间件**：基于我自己给的规模假设（峰值 ingest ≤500/s、供应商 ≤数十、扇出 ≤1:数个），并列出升级 broker 的具体指标信号。
- **D3 仅注册表、不开放任意 URL**：为集中管理密钥 + 天然 SSRF 白名单，接受"接入方多一步登记"的代价。
- **D4 分级重试 + 死信≠日志**：按 destination 打严重等级决定重试上限与告警；死信是可重投的结构化记录，不是只写日志。
- **D5 顺序**：用"同 key + 可交换性"判据重核三场景（修正了最初的直觉误判）；v1 不实现 FIFO，只在注册表标注需求 + 用版本号/LWW 兜底，FIFO 推 v2。
