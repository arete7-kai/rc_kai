# 关键决策记录（DECISIONS）

> 这份文件有两个用途：
> 1. **决策记录**——记下每个关键取舍"选了什么、为什么"，直接支撑作业里"哪些关键决策是你自己做出的，以及原因"。
> 2. **答辩提纲**——评审最可能追问的就是这几条，把"我的结论 + 理由"想清楚，答辩时照着讲即可。

> **一个贯穿全文的设计观察**：D3 的注册表不只是"寻址 + 密钥"，它是整个系统的**策略控制面**——每个目标在注册表里携带自己的重试等级（D4）、是否告警、以及是否需要顺序（D5）。把这些策略下沉到注册表，而不是散在代码里，是本设计的一条主线。

## 第一梯队：定义系统本质（必须亲自拍板）

### D1. 投递语义 = at-least-once

- **默认（我建议）**：at-least-once。先落盘再确认 → 至少一次 → 存在重复 → 要求接收方幂等 / 透传幂等键。
- **为什么重要**：这是全系统地基，评审第一个大概率问"你怎么保证可靠"。关键在于**主动承认"会重复"这个代价**，而不是假装 exactly-once。

**✅ 我的决定**：
at-least-once。

**📝 我的理由**：
需求是"通知必须可靠送达、不关心返回值"，这决定了必须异步接收 + 持续重试。而我不做 exactly-once，是因为**跨第三方根本做不到**——恰好一次要求所有接收方都支持幂等去重，这不在我的控制范围内。所以我选 at-least-once，明确接受"可能重复"这个代价，并用两点兜底：透传业务方的幂等键给供应商去重、在文档中要求接收方做成幂等。

需要澄清一个措辞：发送端并不能保证"投递成功"（对方可能就是不可用）；我能保证的是**意图被持久化，并一直重试到成功或进死信**，绝不悄悄丢失。

### D2. v1 不上消息队列，用"数据库即队列"

- **默认（我建议）**：Postgres + `FOR UPDATE SKIP LOCKED`，不引入 Kafka/RabbitMQ。
- **为什么重要**：这是最能体现"拒绝过度设计"的加分项，但**说服力完全建立在规模假设上**。假设错了，这个决策就错了。

**✅ 我的规模假设**（写进 README）：
预期峰值 ingest ≤ 500 通知/秒、供应商 ≤ 数十个、扇出 ≤ 1:数个。此量级下单实例 Postgres + `SKIP LOCKED` 成立。

**✅ 我的决定**：
Postgres + `FOR UPDATE SKIP LOCKED`，不引入 Kafka/RabbitMQ。

**📝 我的理由**：
- **写放大是真实基准**：一条通知成功路径上至少 3 次 DB 写（INSERT → UPDATE 领取 → UPDATE 完成），有重试再叠加。500 通知/秒 ≈ 1500 写事务/秒，单实例中等规格 Postgres（约 4 vCPU / 8–16GB）在轻松范围内。
- **`SKIP LOCKED` 的正确性与供应商数量无关**：几十个供应商共享一张表没有正确性问题；真正受数量影响的是公平性/队头阻塞，那是上百个目标时才需要处理的（分片，见 §8）。
- **DB-as-queue 的天花板往往不是 CPU，而是表膨胀**：队列表是 UPDATE 密集的，死元组多，autovacuum 一旦跟不上就恶性循环。所以我先用几招延长跑道，而不是一遇压力就升级：给 `(status, next_attempt_at)` 建部分索引、对该表调更激进的 autovacuum、批量领取（`LIMIT N`）、把 `DELIVERED` 行定期归档/删除到冷表、用 `LISTEN/NOTIFY` 减少空轮询。
- **升级到 broker 的触发信号（用指标而非感觉）**：领取延迟（到期→被取走）持续进入秒级；autovacuum 压不住表膨胀且上述手段用满仍无效；DB CPU/IO 持续 >70% 且加不动 worker；或出现**能力型需求**（多消费者订阅、削峰、背压、跨服务解耦）——这条是能力信号不是吞吐数字，常常比吞吐更早触发。

### D3. 目标寻址与密钥模型

- **默认（我建议）**：供应商注册表优先（业务方引用 `destination_id`，密钥集中管理），而非任意 URL 透传。
- **为什么重要**：这条改变**别的团队怎么接入你**，且关系到密钥会不会散落在请求/日志里。属于影响 API 契约的架构决策。

**✅ 我的决定**：
**仅注册表**——业务方只能引用预先登记的 `destination_id`，不开放任意 URL 透传。

**📝 我的理由**：
集中管理带来三个收益：密钥不散落在每个请求和日志里、SSRF 白名单天然成立（目标都是登记过的）、且注册表可以顺带承载每个目标的策略（重试等级、是否告警、是否需顺序——见 D4/D5）。

我明确**接受的代价**：接入方要多做一步"登记"，牺牲了一点接入便利。我认为对一个要长期稳定、涉及外部密钥的中继服务，这个交换是值的。之所以选"仅注册表"而非"注册表为主 + 任意 URL 兜底"，是因为一旦留了任意 URL 的口子，SSRF 白名单和密钥集中管理的好处就漏掉一半，不如从 v1 就守死。

## 第二梯队：需要业务语境才能定（只有你清楚）

### D4. 放弃策略与错误分类

- **默认（我建议）**：`max_attempts`、指数退避 + 抖动、到上限进死信不丢弃；4xx 不重试、5xx/429/超时重试。
- **为什么重要**：取决于"一条通知发不出去伤害多大"，伤害等级决定重试上限和死信要不要告警。

**✅ 我的决定**：
**按目标在注册表里打"严重等级"标签，不同等级对应不同的重试与告警策略**，而不是全局一刀切。示例两档：

| 等级 | 场景 | max_attempts | 退避上限 | 死信告警 |
|---|---|---|---|---|
| 高危 | 付款改状态、扣库存（涉及钱/合规） | ~10 | 到 ~24h | 是 |
| 低危 | 广告回传 | ~5 | 到 ~1h | 否 |

**错误分类**：4xx 默认不重试（重试无意义），5xx / 429 / 超时 / 连接错误可重试。但要对着真实供应商校准边界——例如 401 可能是 token 过期（可恢复）、有的供应商用 200 包一个错误 body，这些例外在接入时按目标标注。

**📝 我的理由**：
- **死信 ≠ 日志，必须分开**：死信是**结构化、可查询、可一键重投**的记录，存在 DB/持久存储里；日志是给人排障看的可观测数据。失败记录只进日志的话就没法重投了。两者都要，但角色不同。死信保留 60 天，到期清理（这也就是"能否最终丢弃"的答案：进死信不等于立刻丢，而是留 60 天等人工处置，到期归档/清除）。
- **等级只管"重试 + 告警"，不与中间件升级挂钩**：是否升级到消息中间件由 D2 的规模/vacuum 信号决定，跟"单条通知的严重等级"是两回事，不把二者绑定，避免逻辑发散。（措辞提醒：Redis 本身不是 MQ，需要的话是 Redis Streams。）

### D5. 是否需要顺序保证

- **默认（我建议）**：v1 不做全局顺序。
- **判据（重要）**：一个场景是否顺序敏感，**不看业务域直觉，看两条**——(1) 事件是否作用在**同一个 key** 上（不同用户/商品/Contact 之间天然无所谓顺序）；(2) 同 key 的操作**是否可交换（commutative）**。

**✅ 我的决定 / 逐场景重判**：

| 场景 | 是否顺序敏感 | 判据 |
|---|---|---|
| 注册回传 | 否，安全 | 一次性事件，不存在同 key 的先后覆盖。 |
| 下单扣库存 | **取决于 API 形态** | 若是"扣减 N 件"这种**相对增量** → 可交换 → 安全；若是"库存**设为** X"这种**绝对赋值** → 不可交换 → 敏感。接入时需确认真实 API 是哪种。 |
| 付款改 CRM 状态 | **敏感** | 这是**状态赋值**（pending→paid→cancelled），非交换。迟到的旧 `pending` 覆盖较新的 `paid` 会出错——这类才是最典型的顺序敏感场景。 |

> 修正说明：我最初凭直觉把"付款"当安全、"库存"当不安全，用上面两条判据重核后发现基本反了——**状态机类的赋值更新才是高危区**，而增量型的库存反而安全。

**v1 的诚实立场**：并行 worker（`SKIP LOCKED`）本身就不保证顺序，真做 FIFO 要对同 key **串行投递**，与并行模型冲突、是实打实的额外复杂度。所以 v1 我这样处理：

1. 注册表里对目标标注"是否需要顺序"，把需求**记录下来**；
2. 真正的 per-key FIFO 实现**推到 v2**（见 README §8）；
3. v1 先用一个更省的逃生舱：**payload 带版本号/时间戳，由接收方做 last-write-wins 丢弃旧值**——这样根本不用实现顺序就能避免"旧覆盖新"。

**📝 我的理由**：
用"同 key + 可交换性"作判据，避免了按业务名称拍脑袋；对少数真敏感的目标，用版本号/LWW 这个低成本手段兜住 v1，把重的 FIFO 留到确有需要时再做，符合"第一版不背额外复杂度"的原则。

## 补充：实现期确认的取舍

### D6. 崩溃回收不计入 attempts

- **默认（我建议）**：租约超时回收（§4.7 的 `ReclaimExpired`）把卡在 `DELIVERING` 的行重置为 `PENDING` 时，**不给 `attempts +1`**。
- **为什么重要**：它决定了"投到上游成功、但在写回 `DELIVERED` 前反复崩溃"这一极端情形下会不会无限重投。

**✅ 我的决定**：
崩溃回收不计入 `attempts`（选项 A）。

**📝 我的理由**：
这次崩溃并没有产生一次"完整的投递尝试结果"，把它计入 `attempts` 会让正常的进程重启/部署也白白消耗重试预算。代价是理论上的重复投递风险——这与 D1 的 at-least-once 一脉相承，由**接收方幂等 + 透传幂等键**兜底。若未来观测到某任务反复被回收，再引入独立的 `reclaim_count` 上限来止损，而不是现在就加。
